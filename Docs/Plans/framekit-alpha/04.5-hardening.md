# 04.5. Endurecimiento del contrato

## Objetivo

Corregir las brechas descubiertas después de verificar las fases 00 a 04 antes
de mover Studio a un workspace o publicar la API. Esta fase sustituye las
reglas históricas indicadas en 00, 02 y 03 cuando exista una contradicción.

La fase 05 no comienza hasta cerrar esta fase.

## Decisiones cerradas de autoría

FrameKit no tiene una lista global de idiomas. Las claves de `content` definen
los locales válidos de cada plantilla y `language` es únicamente su etiqueta
visible:

```tsx
content: {
  es: { language: 'Español', title: 'Hola' },
  fr: { language: 'Français', title: 'Bonjour' },
}
```

En este ejemplo, `locale` es `'es' | 'fr'`. Añadir o retirar una entrada cambia
esa unión automáticamente. Las claves internas distintas de `language` deben
existir en `fields`; un locale nuevo es válido, pero una clave como `titel` no.

Se soportan dos formas de autoría:

1. Plantillas pequeñas definen todo dentro de `defineTemplate` y reciben tipos
   contextuales en `render` sin importar `TemplateRenderProps`.
2. Plantillas complejas separan dimensiones, campos y contenidos mediante
   `defineTemplateBase`, derivan las props del componente y ensamblan el default
   export en `template.tsx`.

`template.tsx` sigue siendo el único punto de descubrimiento. Los demás archivos
y directorios dentro de esa carpeta son implementación privada de la plantilla.

## Definición base tipada

- [ ] Añadir `defineTemplateBase` como función de identidad pública, sin I/O,
  registro global ni estado.
- [ ] Exigir en su argumento `width`, `height`, `fields` y `content`.
- [ ] Inferir las claves concretas de `fields` y de `content` sin ensancharlas a
  `string`.
- [ ] Dar autocompletado dentro de cada locale para `language` y las claves
  declaradas en `fields`.
- [ ] Exigir `language: string` y rechazar claves internas no declaradas en
  `fields`.
- [ ] Permitir que cada locale omita campos. La resolución sigue siendo defaults,
  contenido localizado y ediciones; un requerido que termine vacío se rechaza
  al validar.
- [ ] Aceptar cualquier clave de locale válida como clave de objeto; no introducir
  un tipo global `Locale` ni una lista fija de idiomas.
- [ ] Hacer que `TemplateRenderProps<typeof templateBase>` derive `data`, `locale`,
  `width` y `height` de una definición base completa.
- [ ] Mantener `TemplateRenderProps` como API pública para componentes extraídos,
  pero no exigir su importación para un `render` inline.
- [ ] Hacer que `defineTemplate({ ...templateBase, render })` conserve los tipos
  concretos y produzca una `TemplateDefinition` consumible por el editor.

El patrón complejo documentado será:

```tsx
// definition.ts
import { defineTemplateBase, fields } from '@mauriciodmo/framekit'
import type { TemplateRenderProps } from '@mauriciodmo/framekit'

export const templateBase = defineTemplateBase({
  width: 1080,
  height: 1080,
  fields: {
    title: fields.text({ label: 'Título' }),
    accentColor: fields.color({
      label: 'Color',
      defaultValue: '#b9f8d2',
    }),
  },
  content: {
    es: { language: 'Español', title: 'Hola' },
    fr: { language: 'Français', title: 'Bonjour' },
  },
})

export type ArtworkProps = TemplateRenderProps<typeof templateBase>
```

```tsx
// artwork.tsx
import type { ArtworkProps } from './definition'

export function Artwork({ data, locale, width, height }: ArtworkProps) {
  return (
    <article lang={locale} style={{ width, height, color: data.accentColor }}>
      {data.title}
    </article>
  )
}
```

```tsx
// template.tsx
import { defineTemplate } from '@mauriciodmo/framekit'

import { Artwork } from './artwork'
import { templateBase } from './definition'

export default defineTemplate({
  ...templateBase,
  render: Artwork,
})
```

## Inferencia y dimensiones

- [ ] Rediseñar los genéricos de `TemplateDefinition`, `TemplateRenderProps`,
  `defineTemplateBase` y `defineTemplate` como un solo contrato coherente.
- [ ] Hacer que `render({ data, locale })` inline conozca exactamente las claves de
  `fields` y `content` sin anotación manual.
- [ ] Reemplazar la fixture positiva basada en comentarios por assertions de tipos
  ejecutables por `tsc`.
- [ ] Comprobar con una fixture que dos locales arbitrarios producen exactamente
  `keyof typeof content`; la prueba no debe imponer `es` ni `en` globalmente.
- [ ] Comprobar que una clave de `data` inexistente y una clave interna de
  `content` no declarada en `fields` producen errores de TypeScript esperados.
- [ ] Añadir una fixture del patrón extraído con `defineTemplateBase`,
  `TemplateRenderProps<typeof templateBase>` y un componente en otro archivo.
- [ ] Exigir enteros positivos finitos para `width` y `height` tanto en
  `defineTemplateBase`/`defineTemplate` como en `validateTemplateDefinition`.

## Scanner y carpetas auxiliares

- [ ] Cambiar `findTemplates` para que, al encontrar `template.tsx`, registre esa
  carpeta y deje de recorrerla.
- [ ] Eliminar el error que actualmente interpreta cualquier subdirectorio como
  una plantilla hija.
- [ ] Permitir archivos y carpetas auxiliares como `definition.ts`, `artwork.tsx`,
  `components/`, `helpers/` y `assets/` dentro de una plantilla.
- [ ] No interpretar ni rechazar un `template.tsx` que exista dentro de esas
  carpetas privadas: el límite de recorrido es la primera carpeta reconocida
  como plantilla.
- [ ] Separar la ejecución del script de sus funciones mínimas de scanner y
  generación para poder probarlas contra directorios temporales.

## Validación independiente de la interfaz

- [ ] Cambiar `validateTemplateData` para devolver errores estructurados con los
  códigos `required`, `invalid_number`, `number_too_small`, `number_too_large` e
  `invalid_url`.
- [ ] Incluir `min` o `max` en el error estructurado correspondiente.
- [ ] Eliminar de `validateTemplateData` el parámetro `locale`, actualmente no
  utilizado, y todos los textos en español.
- [ ] Traducir los códigos dentro del editor usando los mensajes de la interfaz y
  sustituir `{min}` o `{max}` allí.
- [ ] Validar cada descriptor cargado en runtime: objeto, `kind` reconocido,
  `label` string no vacío, `placeholder`, `required`, `defaultValue`, límites
  numéricos finitos y `min <= max`.
- [ ] Rechazar `min` y `max` en campos no numéricos y comprobar que `render` sea
  una función.

## Estado y controles del editor

- [ ] En `resetLocale`, copiar `dataByLocale` antes de eliminar el locale para no
  mutar el estado anterior.
- [ ] Limpiar todos los errores con `setErrors({})` al restablecer o cambiar el
  locale de contenido.
- [ ] Copiar `min` y `max` al adaptar un descriptor numérico para `EditorField`;
  no rediseñar el registro de controles solo para este cambio.
- [ ] Al fallar la exportación, localizar el contenedor `data-field-key` y enfocar
  su primer `input`, `textarea` o `select`, no el `div` contenedor.

## Plantilla piloto

- [ ] Convertir `backgroundImage` en un campo con
  `defaultValue: '/images/backgrounds/forest.svg'`.
- [ ] Conservar `accentColor.defaultValue` y retirar ambos valores repetidos de
  `content.es` y `content.en`.
- [ ] Retirar la anotación manual de `TemplateRenderProps` del `render` inline para
  que la plantilla piloto pruebe la inferencia contextual real.
- [ ] Comprobar que defaults, contenido localizado y ediciones producen los mismos
  datos visuales finales.

## Pruebas de regresión

- [ ] Añadir Vitest y los scripts raíz `test` y `typecheck` antes de implementar
  las correcciones no triviales de esta fase.
- [ ] Probar el scanner con carpetas auxiliares y con un `template.tsx` interno que
  debe quedar fuera del recorrido.
- [ ] Probar dimensiones decimales, descriptores inválidos, `render` ausente y cada
  código estructurado de validación.
- [ ] Probar que el reset no muta el estado anterior, conserva otros locales y
  limpia errores; cambiar de locale también limpia errores.
- [ ] Probar que el input numérico recibe `min` y `max` y que el foco cae en el
  control del primer error.
- [ ] Ejecutar las fixtures inline y extraída dentro de `pnpm typecheck`.

## Cierre

```bash
pnpm templates:generate
pnpm lint
pnpm test
pnpm typecheck
pnpm build
```

- [ ] Los cinco comandos terminan correctamente.
- [ ] La plantilla piloto no importa `TemplateRenderProps` para su `render` inline.
- [ ] Una plantilla compleja puede separar definición y componente sin perder
  autocompletado ni aceptar claves desconocidas.
- [ ] Ningún tipo del núcleo depende de los idiomas o mensajes de Studio.
