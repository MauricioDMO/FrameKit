# 09.5. Studio empaquetado

Esta fase extrae la interfaz completa del editor desde `apps/studio/` hacia
`@mauriciodmo/framekit`, para que un proyecto creado no implemente de nuevo la
carga de plantillas, la navegación ni las preferencias visuales.

## API pública

- [x] Exponer `FrameKitStudio` desde `@mauriciodmo/framekit/studio`.
- [x] Exponer `FrameKitStudioRoot` desde
  `@mauriciodmo/framekit/studio/root`.
- [x] Mantener ambas entradas separadas: `FrameKitStudioRoot` usa
  `next/headers` y solo puede entrar en un Server Component; el entry
  `studio` queda libre de APIs de servidor y puede importarse desde una página
  cliente.
- [x] Exponer `frameKitMessages`, `getFrameKitLocale`,
  `FrameKitLocale`, `FrameKitStudioMessages` y `FrameKitStudioTemplate` desde
  el entry cliente cuando corresponda.
- [x] Compilar las dos entradas con `tsdown`, generar sus declaraciones y
  declararlas en `exports`.

## Interfaz reusable

- [x] `FrameKitStudioRoot` resuelve idioma (`es` o `en`) desde la cookie o el
  header `accept-language`, monta el provider de locale y aplica el tema
  persistido antes de hidratar.
- [x] `FrameKitStudio` recibe el arreglo `templates` generado y obtiene el slug
  con `useParams`, evitando serializar loaders de imports dinámicos a través del
  límite Server/Client de Next.
- [x] Derivar la navegación desde el manifiesto de `templates` mediante
  `manifestToNavigation`; aceptar arreglos `readonly` porque el registro
  generado es inmutable para el consumidor.
- [x] Incluir sidebar, navegación, selector de idioma, tema claro/oscuro,
  estado vacío, loading, definición inválida, error de carga y 404 de plantilla.
- [x] Cargar el módulo de plantilla, validar su `default` con
  `validateTemplateDefinition` y montar `FrameKitEditor` solo cuando el estado
  sea válido.
- [x] Conservar el texto del Studio en español e inglés, incluidos los mensajes
  requeridos por `FrameKitEditor`.
- [x] Incluir el árbol `studio/` en el escaneo de Tailwind al generar
  `dist/styles.css`.

## Migración de Studio

- [x] Reemplazar el layout raíz local por `FrameKitStudioRoot`.
- [x] Reducir `src/app/editor/[[...slug]]/page.tsx` a una página cliente que
  entrega `templates` a `FrameKitStudio`.
- [x] Conservar un `editor/layout.tsx` de passthrough mínimo para mantener
  estable el contrato de tipos generado por Next.
- [x] Eliminar de `apps/studio` los componentes locales de sidebar, ruta de
  plantilla, estados visuales, tema e i18n ya provistos por el paquete.
- [x] Mantener la página raíz como redirección a `/editor`.

## Plantilla de create-framekit

- [x] Aplicar el mismo layout y página mínima que Studio:

```tsx
'use client'

import { FrameKitStudio } from '@mauriciodmo/framekit/studio'
import { templates } from '@framekit/generated/templates'

export default function EditorPage() {
  return <FrameKitStudio templates={templates} />
}
```

- [x] Configurar Tailwind CSS 4 con `tailwindcss`, `@tailwindcss/postcss` y
  `postcss` como dependencias de desarrollo.
- [x] Añadir `postcss.config.mjs` con `@tailwindcss/postcss`.
- [x] Importar `tailwindcss` y `@mauriciodmo/framekit/styles.css` desde
  `src/app/globals.css`.
- [x] Eliminar los estilos globales locales que duplicaban los estilos del
  layout empaquetado.
